# GitHub 代码分析器

## 概述

接收GitHub仓库链接，深入分析代码库的核心实现，生成产品经理易懂的Markdown分析报告。

## 使用场景

- 产品经理需要了解竞品的技术实现
- 技术人员需要向产品经理解释代码逻辑
- 团队成员需要快速理解新项目的架构

## 分析流程

### 步骤1: 获取仓库信息

1. 解析GitHub链接，提取owner和repo
2. 使用分析脚本获取基础信息：
   ```bash
   python3 /Users/bytedance/.claude/skills/github-code-analyzer/scripts/analyze_github.py <github_url>
   ```
3. 获取基础信息：目录结构、技术栈、入口文件、API文件

### 步骤2: 深入核心代码分析

根据脚本输出的关键文件，进行深度代码分析：

1. **读取入口文件** - 理解项目启动流程
2. **分析核心模块** - 识别关键类和函数
3. **追踪数据流** - 理解数据如何流转
4. **理解业务逻辑** - 把握核心功能实现

### 步骤3: 生成分析报告

按照以下模板生成Markdown报告。

## 代码分析要点

### 理解代码的提问框架

对于每个核心代码片段，回答：

1. **这段代码做什么？** - 功能描述
2. **为什么这样实现？** - 设计考量
3. **如何被调用？** - 调用关系
4. **数据从哪里来？** - 输入来源
5. **结果送到哪里？** - 输出去向

### 代码解释技巧

- 避免直接贴代码，用自然语言描述
- 使用"如果...那么..."的逻辑描述
- 必要时展示关键代码行并解释
- 区分"什么是API"这类术语并解释

### 产品视角转换

| 技术术语 | 产品语言 |
|----------|----------|
| API接口 | 用户和系统对话的方式 |
| 数据库 | 存储数据的地方 |
| 缓存 | 加快访问速度的临时存储 |
| 异步处理 | 后台慢慢处理，不让用户等 |
| 鉴权 | 验证用户身份 |

---

## 输出要求

1. **语言风格**: 专业但易懂，避免过度技术术语
2. **结构清晰**: 使用Markdown标题层级
3. **代码解释**: 关键代码需附带通俗解释
4. **图表辅助**: 适当使用表格和流程图
5. **总结要点**: 每个部分最后有简短总结

---

## 注意事项

- 克隆仓库使用 `--depth 1` 只获取最新代码
- 优先分析入口文件和核心模块
- 不要逐行解释代码，抓住核心逻辑
- 报告长度根据项目复杂度控制在 2000-5000 字
